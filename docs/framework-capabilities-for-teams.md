# Framework Capabilities For Vertical Teams

This guide is for:

- vertical developers
- product managers
- QA leads
- solution architects

It answers a simple question:

What do we get by building on AppDarta instead of stitching together prompts, scripts, and custom glue?

## If You Are New, Do This First

The public adoption path is simple:

1. install AppDarta Engine
2. create a vertical from `appdarta-vertical-template`
3. run the wizard
4. inspect the project
5. configure business specs and AI roles
6. build and run

This matters because the value of AppDarta is not only in what it can do. It is also in how quickly a team can move from idea to governed vertical workflow.

## The Short Answer

AppDarta gives you a framework-owned control plane for agentic products.

That means your team can focus on business workflows and domain logic while the framework provides:

- lifecycle structure
- runtime control
- policy attachment
- tank-backed knowledge access
- operator-facing execution visibility

Put differently:

- you build the functional piece
- AppDarta handles the technical lifting that usually slows down agentic product teams

The value is:

- you can build a vertical product faster
- you can make it auditable
- you can scale from one agent to many
- you can keep business logic separate from framework plumbing

That separation matters in real buying conversations because technical leaders and product managers can both see where value lands:

- product teams get faster domain iteration
- engineering teams get reusable platform behavior
- QA and operators get clearer runtime and policy visibility

## The Box View

Use this model when evaluating AppDarta.

### The Framework Box

AppDarta owns the heavy lifting:

- typed lifecycle specs
- CLI and project workflow
- gateway pipelines
- orchestration and routing
- single-agent runtime execution
- multi-agent handoff and parallel patterns
- policy hooks and decision surfaces
- Data Tank and context integration
- AI/provider governance and accounting
- operator-facing UI and runtime artifacts

### The Vertical Box

- domain use cases
- business-specific agents
- business-specific policies
- domain tanks and sources
- business thresholds, review paths, and approvals
- business UI modules and operator language

That split is the real power move.

It means you are not rebuilding the platform every time you create a new healthcare, finance, legal, or operations workflow.

It also means AppDarta can be sold as a repeatable framework capability instead of a one-off project accelerator.

## What The Framework Already Enables

### Policy-Aware Gateway

AppDarta includes a separate gateway runtime that can:

- validate and transform requests
- run pre/post handlers
- attach policy checks
- support async callback and approval flows
- expose a cleaner operational boundary than “call the agent directly”

For PMs:

- this creates a real control point for regulated or high-risk workflows

For QA:

- this gives you a testable place to verify approvals, retries, and failure behavior

### Single-Agent And Multi-Agent Patterns

AppDarta is not trapped in a single-agent demo shape.

The framework already supports:

- isolated agent execution
- declarative handoff rules
- specialist-agent routing
- parallel fan-out and gather behavior
- approval and resume checkpoints

For product teams:

- you can start simple and still grow into richer agentic workflows without rewriting the platform

### Data Tanks And Knowledge Retrieval

AppDarta gives verticals a governed knowledge layer.

You can attach:

- public reference material
- enterprise or business-unit knowledge
- client-scoped documents

The framework then provides:

- build and ingest paths
- search and hydrate paths
- runtime tank access
- freshness and storage visibility

For sales and product positioning:

- this is where AppDarta becomes more than “an LLM app”
- it becomes a knowledge-governed operating layer for business decisions

### AI Governance Without Killing Velocity

AppDarta keeps model/provider choices visible and centrally managed.

The framework can own:

- model registry
- role bindings
- fallback chains
- token accounting
- budget visibility

For engineering leaders:

- that reduces uncontrolled provider sprawl

For QA and operators:

- that gives you an attribution path when a workflow needs review

## How Teams Configure LLM Usage

This is one of the places where AppDarta feels different from a typical agent stack.

Instead of scattering provider SDK calls through business code:

- the framework owns provider and model-role configuration
- the vertical declares where AI is needed in business terms
- the framework records cost and usage visibility

So the practical split is:

- framework: provider registry, role bindings, routing, fallback, accounting, budget visibility
- vertical: where AI is used, what business step it supports, what policies and prompts apply

That gives technical teams enough control without losing governance, and it gives stakeholders a clearer story about cost, policy, and operational accountability.

## What Your Vertical Can Extend

You are not boxed in. AppDarta is opinionated about control-plane ownership, not about your business strategy.

A vertical can extend:

- policy logic and thresholds
- gateway orchestration configs
- single-agent flows
- multi-agent flows
- domain-specific tanks and partitions
- business-specific handlers and modules
- approval paths and operator checkpoints

The important thing is that these extensions happen through framework contracts instead of random one-off glue.

## Why This Feels Stronger Than A Typical Agent Stack

Many teams can demo an agent.

Far fewer teams can show:

- how the workflow is governed
- where policy applies
- what data source shaped the answer
- how operator approval works
- how a second or third agent joins the path
- how runtime behavior stays consistent across verticals

That is the AppDarta pitch.

It is not just an agent runtime.

It is a framework for building vertical agentic products with structure, visibility, and room to scale.

## Suggested Audience Message

If you need a clean external message, use this:

AppDarta helps teams ship domain-specific agentic products without rebuilding gateway logic, policy wiring, runtime controls, knowledge plumbing, and operator visibility from scratch. Your team owns the business workflow. AppDarta owns the framework box that makes the workflow governable, extensible, and production-shaped.
